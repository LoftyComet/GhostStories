# 《鬼宅事务所》(Ghost House Agency) - Web 端开发设计文档

## 一、 项目概述与核心设计范式

### 1.1 游戏定位
《鬼宅事务所》是一款 **AI原生（AI-Native）的单人非对称合作解谜游戏**。

### 1.2 核心设计理念
传统非对称合作游戏极度依赖双人社交门槛。本作旨在通过“大语言模型（LLM）+ 有限状态机（FSM）”的双层驱动架构，创造一个拥有独立性格、具备恐慌机制且能与系统深度交互的“虚拟AI队友（调查员）”。游戏规避了传统的视觉惊吓（Jump Scare），将乐趣聚焦于**“机制配合”与“非对称信息交叉验证”**，让单人玩家也能获得高度协同、细思极恐的盲区解谜体验。

---

## 二、 玩家与 AI 的对话界面 (Dialogue Terminal Interface)

### 2.1 界面视觉风格 (UI/UX)
* **整体视觉：** 暗黑情绪极简主义（Dark Moody Minimalism），模拟 2000 年代复古 CRT 电脑终端界面。
* **颜色编码规范：** * 背景色：极低饱和度的深色/纯黑。
  * 终端文字与UI边缘：采用 P1 荧光粉复古绿（经典终端绿 `#00FF00` 或拟真磷光绿 `#33FF00`）。
* **特效实现：** 对话文本及 UI 边缘需通过 CSS/WebGL 叠加轻微的外发光（Glow）与电子束扫描线（Scanlines）效果。边缘区域需应用轻微的球面化畸变，模拟凸起玻璃屏幕的物理质感。

### 2.2 对话交互逻辑
* **输入机制：** 网页端提供双模输入。玩家可键盘输入文本，亦可长按界面中的虚拟“M”键触发 Web RTC 录音，调用语音转文字（STT）接口发送指令。
* **文本渲染流：** 接收到 AI 调查员的响应后，文本必须以打字机流式（Streaming）逐字吐出，并伴随微弱的电子杂音音频。
* **非对称信息呈现：** 对话框内仅显示调查员经过主观加工的“二手信息”，放大玩家在幕后盲区中的想象力空间。

---

## 三、 纸质资料查看系统 (Archives System)

### 3.1 板块功能设计
网页端左下角常驻“放大镜”图标，点击后以模态弹窗（Modal）或分屏侧边栏的形式展开“纸质资料库”。资料库包含：委托人信件、古宅历史文献、辨鬼指南、平面结构草图等。

### 3.2 非对称解谜核心驱动
* **绝对客观 vs 主观偏差：** 纸质资料库提供的是**绝对客观的事实数据**。
* **甄别逻辑：** 玩家无法亲眼看到鬼宅内部，必须将调查员口述的“主观信息”与资料库中的“客观事实”进行交叉验证，以推断调查员是否受到灵异磁场干扰产生了认知偏差。

---

## 四、 复合交互玩法：实体操作与 AIGC 融合

为打破单纯的“文本对白”限制，网页版需引入以下外围交互机制，增强“人机协同”的实体操作感：

### 4.1 实时解围与异步 QTE (Action Synchronization)
* **机制：** 当 AI 调查员在底层遭遇特定物理阻碍（如密码门卡死）时，状态机会向网页端下发阻塞事件。
* **交互：** 网页终端屏幕弹出限时操作界面（如：点击网格切断备用电源）。玩家时限内操作成功，AI 状态机切回正常；操作失败则导致 AI 受创或恐慌逃跑。

### 4.2 调查员状态安抚与资源管理 (Sanity Management)
* **机制：** 引入调查员“理智值/压力值（Sanity/Stress Level）”。高压力状态下，底层状态机会剥夺玩家的部分指挥权（AI 开始无目的乱跑）。
* **交互：** UI 提供“指令干预工具箱”。玩家可消耗有限的能源点点击“发射安抚音频”等按键，强行修正 AI 底层状态机，将其从“恐慌”切回“冷静”。

### 4.3 环境辅助与电量分配滑块 (Environment Control)
* **交互：** 终端设计物理滑块 UI。玩家需在“通讯功率”与“环境电力（为调查员提供紧急照明）”之间动态分配全局电量，实现策略层面的大局观管理。

### 4.4 AIGC 现场图像涌现流 (Image Generation Flow)
* **机制：** 当调查员进入特定关键房间时，LLM 后台异步提取当前场景的描述词（Prompt）。
* **渲染：** 调用图像生成 API，实时在网页端的“次级监视器”中生成一张低饱和度、带有强烈戏剧性侧光的现场静态噪点图片，作为视觉线索。

## 五、 底层鬼宅空间设计 (Grid-Map & Entity System)

### 5.1 网格系统 (Grid-Based Coordinates)
* **结构：** 鬼宅在代码层表现为 2D 拓扑网格地图（如 12x12 矩阵网格）。调查员每移动一格算作“行动一步”。

### 5.2 绿色实体物体 (Valid Environment Targets)
场景中预设可供 AI 识别的实体物体，包含三类静态元数据：空间绝对坐标、物体名称与属性、潜在交互效果。

### 5.3 底层有限状态机 (FSM) 寻路逻辑
包含以下核心状态转换：
* **无目的探索 (Random Exploration)：** 未下发明确目标时随机寻路。
* **定向导航 (Navigate to Target)：** 玩家下发特定命令后，向指定坐标移动。
* **触发场景交互 (Reach Target Area)：** 到达物体网格，执行动作。
* **恐慌逃跑 (Flee from Ghosts)：** 理智值归零时，自动进入强行无序寻路。

### 5.4 步数脉冲触发机制
调查员每行动 **5步**（或触发核心交互），后台自动拦截当前物理状态（坐标、目击物品、理智值），连同玩家最新指令，打包向顶层 AI 发起请求。

## 六、 AI 系统接入规范与技术架构

项目采用 **“LLM 宏观决策 + FSM 微观执行”** 的双层解耦解构模型。

### 6.1 接口通信数据协议 (JSON Schema)
LLM 接入必须通过 Structured Outputs（结构化输出）规范，强制要求大模型返回标准 JSON 报文。

**API 返回 Response 示例：**
```json
{
  "text_response": "行，李先生。我正往东边走呢……不过这地方阴风阵阵的，总觉得那梳妆台怪怪的，我过去瞅瞅，你盯着点对讲机啊。",
  "fsm_command": {
    "action_state": "Navigate_to_Target",
    "target_object": "古董梳妆台",
    "target_coordinates": [2, 8],
    "modify_sanity": -2
  }
}
